## 问题一：EntryAbility 改造

### 背景
默认生成的 `EntryAbility` 继承 `UIAbility`，需要改为继承 `RNAbility` 以初始化 RN 引擎。

### 修改
- 继承自 `RNAbility` 而非 `UIAbility`
- 添加 `getPagePath()` 返回 `'pages/Index'`，RNAbility 会用此路径自动加载页面
- 移除 `onWindowStageCreate`（RNAbility 内部已处理窗口创建和页面加载）
- 保留的 lifecycle 方法添加 `override` + `super` 调用，确保 RN 引擎生命周期正确

### 涉及文件
```
entry/src/main/ets/entryability/EntryAbility.ets
```

### 背景
需要提供一个标准化的 package 工厂函数，供 RN 引擎创建时调用，用于注册自定义 TurboModule/Fabric 组件。

### 修改

- 导出 `createRNPackages(ctx)` 函数，当前返回空数组

### 涉及文件
```
entry/src/main/ets/RNPackagesFactory.ets
```

## 问题三：Index.ets 改为 RNOH 容器页面

### 背景
默认 Hello World 页面需替换为 RNOH 的 `RNApp` 容器，承载 React Native 渲染。

### 修改
- 导入 `RNApp`、`RNOHCoreContext` 等 RNOH 组件
- 通过 `@StorageLink('RNOHCoreContext')` 获取 RNAbility 注入的 core context
- 使用 `RNApp` 组件配置 RN 实例，包括 bundle provider、appKey 等
- `appKey: "AwesomeProject"` 需与 `index.js` 中 `AppRegistry.registerComponent` 的名称一致

### 涉及文件
```
entry/src/main/ets/pages/Index.ets
```

## 问题四：C++ 编译失败 — PackageProvider 签名不匹配

### 错误信息
```
error: out-of-line definition of 'getPackages' does not match any declaration in 'rnoh::PackageProvider'
```

### 根因
RNOH 0.82.30 的 `PackageProvider.h` 声明需要 `Package::Context ctx` 参数：
```cpp
// RNOH/PackageProvider.h
std::vector<std::shared_ptr<Package>> getPackages(Package::Context ctx);
```

但 `PackageProvider.cpp` 中定义的签名不带参数：
```cpp
// 错误：缺少参数
std::vector<std::shared_ptr<Package>> PackageProvider::getPackages() {
```

### 修复
```cpp
// 正确：加上 Package::Context ctx 参数
std::vector<std::shared_ptr<Package>> PackageProvider::getPackages(Package::Context ctx) {
    return {};
}
```

### 涉及文件
```
entry/src/main/cpp/PackageProvider.cpp
```

## 问题五：编译成功后白屏 — RNCSafeAreaProvider 缺失

### 现象
- 构建成功，应用启动
- 日志显示 JS bundle 加载成功
- 白屏，无报错弹窗

### 日志线索
```
Couldn't find ComponentJSIBinder for: RNCSafeAreaProvider
```

### 根因
`App.tsx` 使用 `react-native-safe-area-context` 的 `<SafeAreaProvider>` 包裹应用，这是一个 Fabric 原生组件。RNOH 中没有对应的 ComponentJSIBinder，导致整个组件树渲染失败。

```
App.tsx 组件树:
  <SafeAreaProvider>          ← RNCSafeAreaProvider (Fabric 原生组件，RNOH 不支持)
    <StatusBar />
    <AppContent />
  </SafeAreaProvider>
```

### 修复
移除 `SafeAreaProvider` 依赖，改用 RN 内置的 `<SafeAreaView>`：
```tsx
// 修改前
import { SafeAreaProvider, useSafeAreaInsets } from 'react-native-safe-area-context';
<SafeAreaProvider>
  <StatusBar />
  <AppContent />
</SafeAreaProvider>

// 修改后
import { SafeAreaView } from 'react-native';
<SafeAreaView style={styles.container}>
  <StatusBar />
  <NewAppScreen templateFileName="App.tsx" />
</SafeAreaView>
```

### 涉及文件
```
AwesomeProject/App.tsx
```

## 问题六：Bundle 同步流程

### 背景
修改 RN 源码 (App.tsx) 后，需要重新生成 bundle 并同步到 HarmonyOS 项目。

### 流程
```sh
# Step 1: 生成 bundle
cd AwesomeProject
npm run dev
# 输出到 harmony/entry/src/main/resources/rawfile/bundle.harmony.js

# Step 2: 同步到 TestApplication

   ../TestApplication/entry/src/main/resources/rawfile/bundle.harmony.js

# Step 3: 在 DevEco Studio 中 Rebuild 并运行
```

## 问题七：Metro 热加载 — 虚拟手机连不上 Metro

### 现象
- Metro 在开发机正常运行
- 应用启动后报错：`None of the provided JSBundleProvider was able to load a bundle`
- 虚拟手机上 Metro 连接失败

### 根因
虚拟手机中的 `localhost:8081` 指向的是**设备自身的 8081 端口**，而非开发机的 Metro 服务。Metro 默认绑定 `127.0.0.1`，外部设备无法直接访问。

### 修复
使用 `hdc rport` 将虚拟手机的 8081 端口映射到开发机的 8081 端口：
```sh
hdc rport tcp:8081 tcp:8081
```

映射后，虚拟手机访问 `localhost:8081` 即等同于访问开发机的 Metro 服务。

### 备选方案
如果 `hdc rport` 不可用，也可让 Metro 监听所有网络接口：
```sh
npx react-native start --host 0.0.0.0
```
然后虚拟手机通过开发机的局域网 IP 连接。

## 问题八：rawfile bundle 文件丢失

### 现象
- 先前已成功生成并复制 `bundle.harmony.js` 到 rawfile
- DevEco Studio Rebuild 后 rawfile 目录变空
- 导致 `ResourceJSBundleProvider` 加载失败

### 根因
DevEco Studio 的构建过程（尤其是 Clean Project / Rebuild）会清理 `resources/rawfile/` 目录，手动放置的 bundle 文件被删除。

### 解决方案
每次 rebuild 后需要重新复制 bundle：
```sh
cd AwesomeProject
npm run dev
cp harmony/entry/src/main/resources/rawfile/bundle.harmony.js \
   ../TestApplication/entry/src/main/resources/rawfile/
```

或者完全依赖 Metro 热加载（端口映射正常时无需 rawfile bundle）。

## 三种 Bundle 加载方式

| 方式 | Provider | 适用场景 |
|------|----------|----------|
| Metro 热加载 | `MetroJSBundleProvider()` | 开发调试，修改 JS 后自动刷新 |
| 沙箱目录 hdc 推送 | `FileJSBundleProvider(path)` | 无需 Metro 的快速测试 |
| rawfile 内置 | `ResourceJSBundleProvider(resourceManager, name)` | 正式包 / 离线运行 |

可通过 `AnyJSBundleProvider([...])` 实现优先级兜底链。

## 调试技巧

1. **检查 rnohCoreContext 是否就绪** — 在 Index.ets 的 `aboutToAppear` 中添加 hilog 日志
2. **查看设备日志** — DevEco Studio Logcat 面板搜索 `RNOH` / `reactnative` / `JS`
3. **隔离 bundle 提供方式** — 先用 `ResourceJSBundleProvider` 排除 Metro 连接问题
4. **检查缺失的 TurboModule** — 日志中 `Couldn't provide turbo module "XXX"` 表示需要注册对应模块
5. **检查缺失的 Fabric 组件** — 日志中 `Couldn't find ComponentJSIBinder for: XXX` 表示 RNOH 不支持该原生组件
6. **Metro 连接不上** — 虚拟手机中 `localhost` 指向设备自身而非开发机，需使用 `hdc rport tcp:8081 tcp:8081` 端口映射
7. **rawfile 目录被清空** — DevEco Studio 的 Clean/Rebuild 可能清空 rawfile 目录，离线方案需重新复制 bundle
