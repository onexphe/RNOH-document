# 减少线程间通信 — 架构优化分析

> **需求时间**：2024 年初
> **发布版本**：5.0 / 5.1
> **核心目标**：通过减少不必要的线程切换与跨线程通信开销，提升 RNOH 框架运行时性能

---

## 1. 概述

在 React Native for OpenHarmony（RNOH）中，多线程模型是保障 UI 流畅性的关键。然而，线程间通信（如跨线程 task 投递、同步阻塞调用 `callSync`、消息拷贝）本身存在不可忽视的性能开销。当 TurboModule 运行在非 UI 线程却需要频繁与 UI 线程交互时，线程切换的代价可能抵消甚至超过多线程带来的收益。

"减少线程间通信" 这一需求贯穿了 2024 年多个优化提交，涵盖以下五个具体方向：

| # | 优化项 | 提交时间 | 提交 |
|---|--------|----------|------|
| 1 | ImageLoaderTurboModule 从 Worker 迁移到 UI 线程 | 2024-10 | `9c6d7f207` |
| 2 | BACKGROUND 线程废弃 | 持续（0.72 ~ 0.75） | 多提交 |
| 3 | Mounting 操作在 UI 线程立即执行 | 2024-09 | `449f20ab7` |
| 4 | Fabric Transaction 拉取移到 MAIN 线程 | 2024-09 | `4e363d5db` |
| 5 | 延迟任务在当前线程直接调度 | 2024-08 | `2a78c8105` |

下文逐一分析每个改动的背景、实现细节和性能收益。

---

## 2. ImageLoaderTurboModule 从 Worker 线程迁移到 UI 线程

### 2.1 背景

`ImageLoaderTurboModule` 负责图片的预取、缓存查询和尺寸获取。在 C-API 架构中，它曾被设计为运行在 Worker 线程（`AnyThreadTurboModule`），以期将图片解码等耗时操作从 UI 线程剥离。

然而，`ImageComponentInstance`（运行在 CPP 侧的组件实例）需要与 `ImageLoaderTurboModule` 通信以获取图片资源的预取状态，从而正确选择图片来源。当该 TurboModule 运行在 Worker 线程时，CPP 侧必须通过 `callSync` 跨线程同步调用才能获取 ArkTS 侧的状态：

```
CPP (MAIN)  →  WorkerTurboModule::callSync  →  ArkTS (WORKER)  →  return result
                                                  ↑
                                             线程阻塞等待
```

### 2.2 改动内容

**删除 `ImageLoaderWorkerTurboModule.ts`**（commit `9c6d7f207`）

- 删除了 `ImageLoaderWorkerTurboModule` 类（88 行）
- 从 `Package.ets` 和 `turboModules/index.ts` 中移除对应导出

**修改 `ImageLoaderTurboModule.ts`**

```typescript
// 改动前：继承 TurboModule（Worker 线程）
export class ImageLoaderTurboModule extends TurboModule {
  constructor(protected ctx: TurboModuleContext) { ... }
}
```

```typescript
// 改动后：继承 UITurboModule（UI 线程）
export class ImageLoaderTurboModule extends UITurboModule {
  constructor(protected ctx: UITurboModuleContext) { ... }
}
```

同时新增了 `getPrefetchResult` 方法，直接从同线程的内存缓存中返回结果，无需跨线程调度：

```typescript
protected getPrefetchResult(uri: string): string | undefined {
  return this.imageLoader.getPrefetchResult(uri);
}
```

### 2.3 性能收益

根据提交中的 benchmark 数据，`ImageLoader::getPrefetchResult` 的调用耗时：

| 场景 | 耗时 |
|------|------|
| Worker 线程（跨线程 callSync） | **3.678 ms** |
| MAIN 线程（同线程直接调用） | **0.401 ms** |

**性能提升约 9 倍**，且消除了 `callSync` 对调用线程的阻塞等待。

### 2.4 关键代码路径

`ImageSourceResolver.h:264-284` — `getPrefetchedImageFileUri` 方法通过 `callSync` 调用 `ImageLoaderTurboModule::getPrefetchResult`：

```cpp

    facebook::react::ImageSource imageCandidate) {
  auto rnInstance = m_rnInstance.lock();
  if (rnInstance == nullptr) return {};
  auto imageLoaderTurboModule =
      rnInstance->getTurboModule<ImageLoaderTurboModule>("ImageLoader");
  if (imageLoaderTurboModule != nullptr) {
    auto prefetchResult = imageLoaderTurboModule->callSync(
        "getPrefetchResult", {imageCandidate.uri});
    // ...
  }
  return {};
}
```

`ArkTSTurboModule.cpp:63-98` — `callSync` 的实现使用 `runSyncTask` 跨线程调度并阻塞等待：

```cpp
folly::dynamic ArkTSTurboModule::callSync(
    const std::string& methodName, std::vector<IntermediaryArg> args) {
  // ...
  m_ctx.taskExecutor->runSyncTask(
      m_ctx.turboModuleThread, [ctx = m_ctx, &methodName, &args, &result]() {
        // 在目标线程上执行 ArkTS 调用
      });
  // 调用线程阻塞等待结果
  return result;
}
```

## 3. BACKGROUND 线程废弃

### 3.1 背景

React Native 的 `enableBackgroundExecutor` 功能旨在为布局计算提供一个独立的 BACKGROUND 线程，以减轻 JS 线程的负载。然而在 RNOH 实践中，该特性引入了复杂的线程间通信和状态竞争问题。

### 3.2 废弃标记

**C++ 侧** — `TaskExecutor.h:22-26`

```cpp
enum TaskThread {
  MAIN = 0,
  JS,
  /* @deprecated Use other thread instead —
   * https://github.com/react-native-community/...
   * (latestRNOHVersion: 0.75.0)
   */
  BACKGROUND,
  WORKER,
};
```

**ArkTS 侧** — `RNInstance.ts:499-503`

```typescript
/**
 * UNSTABLE: Enables an additional BACKGROUND thread for layout calculations.
 * 增加死锁和崩溃风险。通常因 "Assertion failed: parser was already assigned" 错误崩溃。
 * @deprecated (latestRNOHVersion: 0.72.27)
 */
enableBackgroundExecutor?: boolean;
```

### 3.3 根源分析

BACKGROUND 线程的通信问题在文档中有明确记录（`docs/zh-cn/01-设计/架构介绍.md:94`）：

> 由于开启 **BACKGROUND** 涉及复杂的线程间通信，在稳定性方面带来风险，因此正式商用版本中不要开启 **BACKGROUND** 线程。

`ShadowTree` 在各个线程间的状态同步需要大量跨线程断言和锁保护。典型的崩溃模式是 `"Assertion failed: parser was already assigned"`，表明 ShadowNode 的解析状态在多个线程间出现了竞争。

## 4. Mounting 操作在 UI 线程立即执行

### 4.1 背景

当 Reanimated 等第三方库在 UI 线程直接触发 Fabric 更新时，`SchedulerDelegate` 的各个回调方法（`schedulerDidDispatchCommand`、`schedulerDidSetIsJSResponder`、`schedulerDidSendAccessibilityEvent` 等）也会在 UI 线程被调用。

在优化前，这些方法统一通过 `performOnMainThread` 向 MAIN 线程投递 task。当调用者已经在 MAIN 线程上时，这会导致不必要的 task 入队和延迟执行。

### 4.2 改动内容

**`SchedulerDelegate.h`** — 在 `performOnMainThread` 模板方法中增加了"已在 MAIN 线程则立即执行"的短路判断：

```cpp
template <typename Operation>
void performOnMainThread(Operation operation) {
  if (m_taskExecutor->isOnTaskThread(TaskThread::MAIN)) {
    if (auto mountingManager = m_mountingManager.lock()) {
      operation(mountingManager);
    }
    return;  // 直接执行，跳过 task 调度
  }
  // 不在 MAIN 线程，通过 task 调度
  m_taskExecutor->runTask(
      TaskThread::MAIN,
      [weakMountingManager = m_mountingManager,
       operation = std::move(operation)] {
        if (auto mountingManager = weakMountingManager.lock()) {
          operation(mountingManager);
        }
      });
}
```

### 4.3 影响范围

此优化影响 `SchedulerDelegate` 的所有回调方法，包括：

- `schedulerDidDispatchCommand` — 派发组件命令
- `schedulerDidSetIsJSResponder` — 设置 JS 响应者
- `schedulerDidSendAccessibilityEvent` — 发送无障碍事件
- `performTransaction` 中的 `willMount` / `doMount` / `didMount` 回调

对于 Reanimated 等频繁在 UI 线程操作 Fabric 的场景，每个操作节省一次完整的 task 调度 + 执行周期（通常数百微秒到数毫秒）。

## 5. Fabric Transaction 拉取移到 MAIN 线程

### 5.1 背景

在旧实现中，`schedulerDidFinishTransaction` 在 JS 线程上被调用，完整的 `pullTransaction` 流程在 JS 线程上发起，再通过三次 `performOnMainThread` 分别将 `willMount`、`doMount`、`didMount` 阶段投递到 MAIN 线程。

这意味着：
1. `transaction` 对象需要在 `pullTransaction` 的多个回调之间**多次拷贝**
2. 三个 mounting 阶段通过**三个独立 task** 投递，增加了调度开销
3. `pullTransaction` 本身在 JS 线程上执行，影响 JS 线程响应性

### 5.2 改动内容

**`SchedulerDelegate.cpp`** — 将整个 `pullTransaction` 移到 MAIN 线程执行：

```cpp
// 改动后：整个 pullTransaction 在 MAIN 线程执行
performOnMainThread([mountingCoordinator](auto mountingManager) {
  mountingCoordinator->getTelemetryController().pullTransaction(
      [&mountingManager](auto const& transaction, ...) {
        mountingManager->willMount(transaction.getMutations());
      },
      [&mountingManager](auto const& transaction, ...) {
        mountingManager->doMount(transaction.getMutations());
      },
      [&mountingManager](auto const& transaction, ...) {
        mountingManager->didMount(transaction.getMutations());
        logTransactionTelemetryMarkers(transaction);
      });
});
```

### 5.3 优化效果

1. **消除 transaction 对象多次拷贝** — `pullTransaction` 的三个回调共享同一个 `mountingCoordinator` 引用，transaction 无需在 JS 和 MAIN 线程之间传递
2. **单一 task 替代多个 task** — 原本需要 3 次 `performOnMainThread`（3 次 task 调度），现在合并为 1 次
3. **JS 线程负载减轻** — `pullTransaction` 的元数据解析和 telemetry 处理不再占用 JS 线程时间

## 6. 延迟任务调度优化

### 6.1 背景

`EventLoopTaskRunner` 的 `runDelayedTask` 方法在创建定时器任务时，**无论调用线程是否就是事件循环所在线程**，都统一通过 `runAsyncTask` 跨线程投递创建操作。

当调用者已经在正确的线程上时，这意味着一次无意义的线程间通信（task 入队 → uv_async_send → 事件循环读取 → 执行）。

### 6.2 改动内容

**`EventLoopTaskRunner.cpp`** — 在创建定时器之前检查当前线程：

```cpp
EventLoopTaskRunner::DelayedTaskId EventLoopTaskRunner::runDelayedTask(
    Task&& task, uint64_t delayMs, uint64_t repeatMs) {
  auto id = m_nextTaskId++;
  // 将 task 封装到 timerCallback 中
  auto timerCallback = [this, id, repeatMs, task = std::move(task)]() mutable {
    task();
    if (repeatMs == 0) {
      m_timerByTaskId.erase(id);
    }
  };

  auto createTimer = [this, id, delayMs, repeatMs,
                      timerCallback = std::move(timerCallback)]() mutable {
    auto [it, inserted] = m_timerByTaskId.emplace(
        id, uv::Timer(m_loop, std::move(timerCallback), delayMs, repeatMs));
  };

  if (isOnCurrentThread()) {
    createTimer();  // 当前线程直接创建定时器
  } else {
    runAsyncTask(std::move(createTimer));  // 跨线程投递
  }
  return id;
}
```

此外，`m_nextTaskId` 从 `uint64_t` 改为 `std::atomic<uint64_t>`，以支持可能的跨线程并发访问。

### 6.3 优化效果

- 当延迟任务在**当前线程**上调度时，完全消除一次 `runAsyncTask` 的通信开销
- 在典型场景中（如定时器、动画帧调度等高频延迟任务），每次调度节省数十到数百微秒

---

## 7. 总结

"减少线程间通信" 这一需求在 2024 年的 RNOH 代码中体现为一组有针对性的优化措施。其核心思路可以概括为：

| 策略 | 实现方式 | 示例 |
|------|----------|------|
| **合并线程归属** | 将需要频繁跨线程交互的模块移到同一线程 | ImageLoaderTurboModule: Worker → UI |
| **消除不必要的线程** | 废弃收益不抵风险的独立线程 | BACKGROUND 线程废弃 |
| **短路判断** | 已在目标线程时直接执行，跳过 task 调度 | `performOnMainThread` + `isOnTaskThread` |
| **批量调度** | 将多次跨线程投递合并为一次 | `pullTransaction` 整体移到 MAIN 线程 |
| **避免间接投递** | 在正确的线程上直接创建资源 | `runDelayedTask` 直接创建定时器 |

这五项优化的共同目标都是在**不改变功能语义**的前提下，尽可能减少线程间通信的次数和代价，从而提升整体框架性能。其中 ImageLoaderTurboModule 的迁移效果最为显著，性能提升约 9 倍。
