## 一、改动详解

### 1. FlashList 并行渲染禁用（核心修复）

**提交**：`8c1ed3ff8` — "fix:When there is a flashlist on the surface, disable the parallelization of instructions."

**改动文件**：`SchedulerDelegate.cpp`

**改动内容**：

| 新增函数/结构 | 作用 |
|---|---|
| `isFlashListSerialOnlyComponent()` | 检测组件名是否为 `AutoLayoutView` 或 `CellContainer`（这两个是 FlashList 的核心布局容器） |
| `shouldForceSerialForSurface()` | 遍历当前事务的所有 mutation，若包含 FlashList 组件则将该 SurfaceId 记录进 `g_surfaceSerialForce` 缓存，后续同一 Surface 的事务也强制串行 |
| `g_surfaceSerialForce` | `unordered_map<SurfaceId, time_point>`，记录哪些 Surface 含 FlashList，带 TTL（10分钟）和容量上限（128）自动淘汰 |
| `pruneSurfaceSerialForceLocked()` | 超量时按 TTL 清理过期 Surface 记录 |

**关键逻辑变更**：

```cpp
// 之前：并行渲染无条件开启
if (IsParallelizationEnabled()) { ... }

// 之后：检测到 FlashList 时强制串行
const bool forceSerialForSurface =
    shouldForceSerialForSurface(surfaceId, transaction->getMutations());
if (IsParallelizationEnabled() && !forceSerialForSurface) { ... }
```

**设计意图**：FlashList 的 `AutoLayoutView` 和 `CellContainer` 对布局时序敏感，并行 Create 会打乱其内部计算的偏移顺序，导致列表项位置跳跃。

---

### 2. 防止 Fabric Create Mutation 被不安全重排

**提交**：`06b1b7794` / `b79148ba1` — "Fix: Prevent unsafe reordering of Fabric Create mutations"

**改动文件**：`SchedulerDelegate.cpp`

**改动内容**：

引入 `CreateBuckets` 结构和 `bucketizeCreateMutations()` 函数，将事务中的 mutations 分为两类：

| 分类 | 含义 |
|---|---|
| `movableCreates` | 可以安全提前执行的 Create mutation（其 tag 在同一事务中没有更早的非 Create mutation 依赖它） |
| `deferredTail` | 必须保持原始顺序的 mutation（包括所有非 Create mutation，以及被非 Create mutation 先引用的 Create） |

**之前的问题**：

```cpp
// 旧逻辑：简单地把所有 Create 和非 Create 分开
for (auto& mutation : mutations) {
  if (mutation.type == Create) {
    sliceableMutations.push_back(mutation);  // 全部可并行
  } else {
    otherMutations.push_back(mutation);      // 全部串行
  }
}
```

这种粗暴分离会破坏 Fabric 的依赖顺序。例如事务中 `Update(tag=5)` 在 `Create(tag=5)` 之前出现时，如果 Create 被提前并行执行，Update 可能操作一个尚未完成 Create 的节点。

**新增安全检查**：

```cpp
// 如果 deferredTail 中包含 Create，则不再分两批渲染
const bool hasCreateInDeferredTail = containsCreateMutation(otherMutationList);
auto split = hasCreateInDeferredTail
    ? TwoBatchSplit{false, otherMutationList, {}}
    : splitNonCreateMutationsIntoTwoBatches(otherMutationList);
```

**修复 job_partner 引用捕获**：将 `[this, &mutation]` 改为 `[this, mutation]`（值捕获），避免并行任务访问已移动的引用。

---

### 3. ScrollView 滚动跳跃修复（mvcp 场景）

**提交**：`7721c456d` — "Fix: Scrolling component with mvcp enabled, normal scrolling exhibits jumping position issue"

**改动文件**：`ScrollViewComponentInstance.cpp/h`

**改动内容**：

新增 `m_contentSizeChanged` 标记，`adjustVisibleContentPosition()` 中的 `scrollTo` 修正仅在 contentSize 真正变化时触发：

```cpp
// 之前：任何 delta != 0 都触发 scrollTo 修正
if (deltaY != 0) {
  m_scrollNode.scrollTo(...);
}

// 之后：只在 contentSize 变化时修正
m_contentSizeChanged = m_contentSize != stateData.getContentSize();
if (deltaY != 0 && m_contentSizeChanged) {
  m_scrollNode.scrollTo(...);
}
```

**问题根因**：`adjustVisibleContentPosition` 本意是在列表内容尺寸变化时修正滚动偏移，但 FlashList 滑动时会频繁触发 `onStateChanged`（即使 contentSize 未变），导致每帧都执行 `scrollTo` 修正 → 视口位置被反复强制调整 → 表现为跳动。

---

### 4. ScrollView horizontal prop 意外复用修复

**提交**：`5e7327d97` — "Fix: ScrollView horizontal prop being unintentionally reused when not explicitly provided (RNOH)"

**改动文件**：`react-native.patch`（JS 层 ScrollView.js）

**改动内容**：在 ScrollView 的 props 序列化中显式传递 `horizontal` 字段：

```js
// 之前：horizontal 未被显式传入 native props
const props = {
  ...otherProps,
  alwaysBounceHorizontal,
  alwaysBounceVertical,
  ...
};

// 之后：显式传入
const props = {
  ...otherProps,
  horizontal,       // <-- 新增
  alwaysBounceHorizontal,
  alwaysBounceVertical,
  ...
};
```

**问题根因**：当 `horizontal` 未显式提供时，Fabric 会复用上一次的 props 值。如果一个竖向 ScrollView 之前渲染过横向的，`horizontal=true` 被缓存复用 → ScrollView 方向错误 → 列表布局错乱 → 可能表现为白屏（内容渲染到错误方向视口外）。

## 二、FlashList 滑动异常跳动 → 白屏的根因分析

### 现象描述

FlashList 在滑动时出现异常跳动，快速跳动导致白屏。

### 根因链路

```
FlashList 滑动
  → 大量 RecyclerItem 更新事务（AutoLayoutView/CellContainer 的 Create + Update + Remove + Insert）
  → 旧代码并行处理 Create mutations（ffrt job_partner）
      ├── 问题 A：Create 被不安全重排，Update/Insert 可能操作未完成 Create 的节点
      ├── 问题 B：FlashList 的 AutoLayoutView 依赖严格顺序计算偏移，并行打乱顺序
      └── 问题 C：job_partner 按引用捕获 mutation，并行任务访问已移动数据
  → 组件树状态不一致
  → adjustVisibleContentPosition 每帧触发 scrollTo 修正（即使 contentSize 未变）
  → 视口反复跳跃
  → 快速跳动时 ArkUI 渲染管线来不及完成帧 → 显示空白帧 → 白屏
```

### 修复映射

| 根因 | 修复提交 | 修复方式 |
|---|---|---|
| **并行渲染打乱 FlashList 布局时序**（问题 A+B） | `8c1ed3ff8` | 检测 FlashList 组件 → 强制该 Surface 串行渲染 |
| **Create 被不安全重排导致依赖断裂**（问题 A） | `06b1b7794` | `bucketizeCreateMutations()` 保证 Create 依赖顺序正确 |
| **job_partner 引用捕获**（问题 C） | `06b1b7794` | 改为值捕获 `[this, mutation]` |
| **scrollTo 修正每帧触发导致跳动** | `7721c456d` | 增加 `m_contentSizeChanged` 条件，仅在 contentSize 真正变化时修正 |
| **horizontal prop 复用导致方向错乱** | `5e7327d97` | 显式传递 `horizontal` 到 native props |

### 最核心修复

**`8c1ed3ff8`（FlashList 串行渲染）** 是直接针对"FlashList 滑动跳动→白屏"的核心修复。它从根本上解决了并行渲染与 FlashList 布局时序的矛盾——当检测到 Surface 上存在 `AutoLayoutView` 或 `CellContainer` 时，整个 Surface 的事务切换为串行执行，避免了 Create 并行导致的偏移计算错乱。

**`7721c456d`（scrollTo 修正条件）** 是辅助修复。即使串行渲染后，`adjustVisibleContentPosition` 的无条件 `scrollTo` 修正仍会在 FlashList 高频更新时引发跳动，增加 `m_contentSizeChanged` 条件后消除了这部分不必要的视口修正。

**`06b1b7794`（Create 重排安全化）** 是基础设施修复，对所有含 Create 的事务提供安全保障，不仅影响 FlashList 也影响其他组件的渲染正确性。

### 修复效果总结

三个修复协同作用：

1. **串行渲染**（`8c1ed3ff8`）→ 防止 FlashList mutation 乱序执行
2. **Create 重排安全化**（`06b1b7794`）→ 防止依赖断裂和引用捕获 bug
3. **scrollTo 修正条件化**（`7721c456d`）→ 防止高频更新时的视口跳跃

三者共同消除了 FlashList 滑动时的跳动和白屏现象。
