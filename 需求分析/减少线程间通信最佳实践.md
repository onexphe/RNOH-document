# 基于线程归属优化的减少线程间通信设计

更新时间: 2026-06-09

---

## 一、概述

### RNOH 多线程通信模型简介

React Native for OpenHarmony（RNOH）采用多线程模型运行应用，核心线程包括 **MAIN**（UI 渲染）、**JS**（JavaScript 执行）、**WORKER**（通用计算）三种。线程间通过 `TaskExecutor` 进行 task 投递和 `runSyncTask` 同步阻塞调用。

当操作频繁需要跨线程协调时，通信本身的开销（task 入队 → uv_async_send → 事件循环读取 → 执行）可能远大于被调度操作的实际工作量。"减少线程间通信"需求的本质不是"减少线程数"，而是**让每个操作在正确的线程上完成，避免不必要的跨线程跳转**。

RNOH 通过 `EventLoopTaskRunner`（基于 libuv 事件循环）统一管理跨线程调度，支持 `runAsyncTask`（异步投递）、`runSyncTask`（同步阻塞）、`runDelayedTask`（延迟执行）三种调度方式，并提供 `isOnCurrentThread()` / `isOnTaskThread()` 线程判断能力。

通常，用"短路执行"指代"已在目标线程则直接调用"的优化模式，用"线程归属"指代"TurboModule 应运行在与主要调用者同一线程"的设计原则。

---

## 二、特性介绍

减少线程间通信是 RNOH 运行时性能优化的核心策略，按需调整操作的执行线程，优化跨线程通信开销。

> **说明**
> 适配减少线程间通信策略的好处包括：消除不必要的 task 投递开销，降低同步阻塞调用对调用线程的冻结时间，减少跨线程数据拷贝，提升高频交互场景的整体响应速度。

---

## 三、场景策略建议

减少线程间通信的核心原则是"让操作在正确的线程上完成"，而非盲目减少线程数量。

### ❌ 不建议将 TurboModule 盲目分配到 Worker 线程

不建议将所有 TurboModule 都设置为 `WorkerTurboModule` 或 `AnyThreadTurboModule`，期望通过"耗时操作剥离 UI 线程"来提升性能。这会在 TurboModule 与 UI 线程频繁交互时引入大量 `callSync` 跨线程阻塞调用。

主要原因有以下几点：

1. **跨线程阻塞冻结调用线程**：`callSync` 通过 `runSyncTask` 在目标线程执行 ArkTS 方法，调用线程阻塞等待结果返回。高频调用下，UI 线程被反复冻结，反而降低响应性。
2. **线程间通信开销远超操作本身**：ImageLoader 的 `getPrefetchResult` 在 Worker 线程耗时 3.678 ms，移到 UI 线程后仅 0.401 ms，9 倍差距完全来自跨线程通信开销。
3. **TurboModule 线程归属应由"与谁通信最频繁"决定**：如果一个 TM 的主要调用者运行在 UI 线程（如 ImageLoader 被 MAIN 线程的 ImageComponentInstance 调用），TM 就应该运行在 UI 线程。

### ✅ 策略建议

| 类型 | 类型描述 | 参数建议（案例） |
|------|----------|------------------|
| **UI 交互型 TurboModule** | 主要调用者在 MAIN 线程，需与 UI 组件频繁交互 | ImageLoader：继承 `UITurboModule`，同线程 `callSync` 短路执行<br>SafeArea：继承 `UITurboModule`，缓存 insets 减少跨线程调用 |
| **独立计算型 TurboModule** | 不依赖 UI 线程 API，主要在 JS 线程被调用 | Networking/WebSocket：继承 `AnyThreadTurboModule`，移至 Worker 线程<br>Blob/FileReader：继承 `AnyThreadTurboModule`，移至 Worker 线程 |
| **Fabric Mounting 操作** | 在 MAIN 线程执行的渲染管线操作 | `performOnMainThread`：增加 `isOnTaskThread(MAIN)` 短路判断<br>`pullTransaction`：整体移至 MAIN 线程，零跨线程通信完成 |
| **延迟任务调度** | 需要在指定线程延迟执行的操作 | `runDelayedTask`：`isOnCurrentThread()` 判断，当前线程直接创建定时器 |

## 四、性能诊断工具

### callSync 超时警告

`ArkTSTurboModule::callSync` 内置了计时逻辑，当单次 `callSync` 超过 2 ms 时输出 WARNING 日志，用于识别不应跨线程运行的 TurboModule。

> 具体操作：查看日志中 `ArkTSTurboModule::callSync: execution time — X ms (TurboModule::method)` 格式的 WARNING，X > 2 的调用即为优化候选。

关键代码（`ArkTSTurboModule.cpp`）：

```cpp
auto start = std::chrono::high_resolution_clock::now();
m_ctx.taskExecutor->runSyncTask(
    m_ctx.turboModuleThread, [&]() { /* ArkTS 调用 */ });
auto stop = std::chrono::high_resolution_clock::now();
auto duration = std::chrono::duration_cast<std::chrono::milliseconds>(stop - start);
if (duration.count() > 2) {
  LOG(WARNING) << "ArkTSTurboModule::callSync: execution time — "
               << duration.count() << " ms (" + this->name_ + "::" + methodName + ")";
}
```

### TraceSection 线程追踪

**步骤1：添加 TraceSection**

在 `callSync` 和 `runSyncTask` 的关键路径中添加 `TraceSection` 标记。

**步骤2：使用 DevTools 抓取 Trace**

通过 HarmonyOS DevTools 的 Performance 面板抓取应用运行时的 trace 数据。

**步骤3：分析跨线程调用频率**

在 trace 数据中筛选 `callSync` / `runSyncTask` 相关的 TraceSection，统计跨线程调用频率和耗时分布。

**步骤4：识别优化目标**

高频且耗时的跨线程调用即为优化目标，考虑将其移至调用者所在线程或添加短路判断。

## 五、使用场景

### 场景说明

在 RNOH 多线程架构下，基于 `EventLoopTaskRunner` 的 `isOnCurrentThread()` / `isOnTaskThread()` 判断能力和 `TurboModule` 三层分类体系（UITurboModule / AnyThreadTurboModule / WorkerTurboModule），可以达到减少不必要的跨线程通信、提升运行时性能的目标。RNOH 支持线程归属优化和短路执行两种核心能力，开发者通过合理分配 TurboModule 线程归属和使用短路判断模式，可以享受减少线程间通信带来的性能收益。

减少线程间通信支持开发者自定义业务维度，其常见使用场景：

- 通过 TurboModule 线程归属优化，用于减少频繁跨线程交互模块的 `callSync` 开销，具体可见本文第六、七章案例。
- 通过短路执行模式，用于消除"已在目标线程却仍投递 task"的无意义通信，具体可见 `EventLoopTaskRunner` 和 `performOnMainThread` 实现。
- 通过结果缓存，用于减少重复的跨线程同步调用，具体可见 SafeAreaView insets 缓存案例。

基于以上使用场景，结合 RNOH 实际演进历程，本文选取**两个**有代表性的案例展开介绍。

> **说明**
> TurboModule 的线程归属不应由"是否耗时"决定，而应由"与谁通信最频繁"决定。BACKGROUND 线程已废弃，不要开启 `enableBackgroundExecutor`。

## 六、基于线程归属优化实现 ImageLoaderTurboModule 迁移

### 效果展示

ImageLoaderTurboModule 从 Worker 线程迁移到 UI 线程后，`getPrefetchResult` 跨线程调用开销从 3.678 ms 降低到 0.401 ms。

### 性能对比

> **说明**
> 对比方法：通过 `ArkTSTurboModule::callSync` 内置计时逻辑测量单次调用耗时。

参考第四章"callSync 超时警告"诊断工具。

分别在 Worker 线程（跨线程 callSync）和 MAIN 线程（同线程直接调用）场景下测量 `ImageLoader::getPrefetchResult` 耗时。

| 场景 | getPrefetchResult 耗时 |
|------|------------------------|
| Worker 线程（跨线程 callSync） | 3.678 ms |
| MAIN 线程（同线程直接调用） | 0.401 ms |
| **性能提升倍数** | **~9x** |

## 七、基于短路执行与管线合并实现 Fabric Mounting 优化

### 效果展示

Fabric `pullTransaction` 从 JS 线程移至 MAIN 线程后，整个 mounting 管线在 MAIN 线程上零跨线程通信完成。`performOnMainThread` 的短路判断使已处于 MAIN 线程的回调直接执行。

### 性能对比

> **说明**
> 对比方法：通过 TraceSection 分析跨线程 task 投递次数和耗时。

优化前：`pullTransaction` 在 JS 线程发起，三个回调（willMount / doMount / didMount）分别通过 `performOnMainThread` 投递到 MAIN 线程，transaction 对象多次拷贝，3 次独立 task 调度。

优化后：`pullTransaction` 整体在 MAIN 线程执行，三个回调直接在 MAIN 线程调用，加上短路判断，零跨线程通信完成。

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 跨线程 task 投递次数 | 3 次 | 0 次（短路时）或 1 次（非短路时） |
| transaction 对象拷贝 | 多次 | 无 |
| JS 线程负载 | 承担 pullTransaction 元数据解析 | 完全释放 |

### 代码实现

**短路判断 — `performOnMainThread` 模板方法**

```cpp
template <typename Operation>
void performOnMainThread(Operation operation) {
  if (m_taskExecutor->isOnTaskThread(TaskThread::MAIN)) {
    if (auto mountingManager = m_mountingManager.lock()) {
      operation(mountingManager);     // ← 直接执行
    }
    return;
  }
  m_taskExecutor->runTask(TaskThread::MAIN,
      [weakMountingManager = m_mountingManager, operation = std::move(operation)] {
        if (auto mountingManager = weakMountingManager.lock()) {
          operation(mountingManager); // ← 异步执行
        }
      });
}
```

`SchedulerDelegate.h:86-102`

**管线合并 — `pullTransaction` 整体移至 MAIN 线程**

```cpp
performOnMainThread([mountingCoordinator](auto mountingManager) {
  mountingCoordinator->getTelemetryController().pullTransaction(
      [&mountingManager](auto const& transaction, ...) {
        mountingManager->willMount(transaction.getMutations());
      },
      [&mountingManager](auto const& transaction, ...) {
        mountingManager->doMount(transaction.getMutations());
      },
      [&mountingManager](auto const& transaction, ...) {
        mountingManager->didMount(transaction.getMutations());
        logTransactionTelemetryMarkers(transaction);
      });
});
```

`SchedulerDelegate.cpp`

> **说明**
> 短路判断和管线合并的组合效果：当调用者已在 MAIN 线程（如 Reanimated 触发的 Fabric 更新），整个 mounting 管线从 `pullTransaction` 到 `didMount` 全程零跨线程通信。延迟任务调度（`runDelayedTask`）也采用了相同的短路模式。

## 八、总结

综上所述，使用减少线程间通信策略可以消除不必要的 task 投递开销，降低同步阻塞调用对线程的冻结时间，减少跨线程数据拷贝。开发时推荐使用以下六项策略，享受运行时性能优化效果：

| 策略 | 模式 | 典型实现 |
|------|------|----------|
| **短路执行** | 已在目标线程 → 直接调用，跳过 task 投递 | `isOnCurrentThread()` / `isOnTaskThread()` 检查 |
| **线程归属优化** | 将 TurboModule 放在与主要调用者同一线程 | ImageLoader → UI; Networking → Worker |
| **C++ 替代 ArkTS** | 将原本需要 JS↔ArkTS 跨线程的操作迁至 C++ 单线程 | TimingTurboModule → HarmonyTimerRegistry |
| **消除线程** | 废弃因线程间通信导致稳定性问题的线程 | BACKGROUND 线程废弃 |
| **管线合并** | 将多步跨线程投递合并为单步在目标线程上执行 | pullTransaction 整体移至 MAIN 线程 |
| **结果缓存** | 缓存跨线程调用结果，避免重复 callSync | SafeAreaView insets 缓存 |

### 短路执行基础设施

```cpp
// EventLoopTaskRunner.cpp — runSyncTask 短路判断
void EventLoopTaskRunner::runSyncTask(Task&& task) {
  if (isOnCurrentThread()) {
    task();         // ← 已在目标线程，直接执行
    return;
  }
  waitForSyncTask(std::move(task));  // ← 跨线程阻塞等待
}

// EventLoopTaskRunner.cpp — runDelayedTask 短路判断
DelayedTaskId EventLoopTaskRunner::runDelayedTask(...) {
  auto createTimer = [...]() mutable { ... };
  if (isOnCurrentThread()) {
    createTimer();                    // ← 直接创建，不走 uv_async
  } else {
    runAsyncTask(std::move(createTimer));  // ← 跨线程投递
  }
}
```
