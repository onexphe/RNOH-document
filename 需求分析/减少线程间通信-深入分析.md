# 减少线程间通信 — 深入分析文档

> **需求时间**：2024 年初启动，持续演进至 2026 年
> **发布版本**：5.0 / 5.1 及后续版本
> **核心目标**：通过减少线程间通信（task 投递、同步阻塞、消息拷贝）提升运行时性能

---

## 1. 问题本质

RNOH 采用多线程模型（MAIN / JS / WORKER / BACKGROUND），线程间通过 `TaskExecutor` 进行 task 投递和 `runSyncTask` 同步阻塞调用。当操作频繁需要跨线程协调时，通信本身的开销（入队 → uv_async_send → 事件循环读取 → 执行）可能远大于被调度操作的实际工作量。

这一需求的本质不是"减少线程数"，而是**让每个操作在正确的线程上完成，避免不必要的跨线程跳转**。

---

## 2. 演进脉络：五个阶段

### Phase 1 — 基础设施建设（2023 Q3 ~ 2024 Q1）

#### 2.1 libuv 事件循环驱动的 TaskExecutor

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `ec951245a` | 2024-07-01 | Juliusz Wajgelt | 引入 `EventLoopTaskRunner`，基于 libuv 替代旧的 task 调度机制 |
| `ef4cdee77` | 2024-07-02 | Juliusz Wajgelt | 延迟任务逻辑从各 TaskRunner 收归到 `EventLoopTaskRunner` |
| `3ff649446` | 2024-07-03 | Juliusz Wajgelt | 重写 `TimingTurboModule`，使用 `runDelayedTask` 替代 ArkTS 定时器 |

**架构解读**：这是所有后续优化的地基。旧调度机制在每次跨线程投递时依赖轮询，`EventLoopTaskRunner` 使用 `uv_async_t` 实现高效唤醒，`uv_timer_t` 实现延迟任务。

**关键代码**（`EventLoopTaskRunner.cpp:53-58`）：

```cpp
void EventLoopTaskRunner::runSyncTask(Task&& task) {
  if (isOnCurrentThread()) {
    task();         // ← 已在目标线程，直接执行
    return;
  }
  waitForSyncTask(std::move(task));  // ← 跨线程阻塞等待
}
```

这个 `isOnCurrentThread()` 短路判断是整个"减少线程间通信"的基石模式。

**TimingTurboModule 重写的意义**：定时器管理从 ArkTS 侧完全迁至 C++ 侧，消除了 JS → ArkTS → JS 的跨线程循环。原先每帧都需要 JS 线程通知 ArkTS 线程设置定时器、ArkTS 线程到期后再回调 JS 线程；现在定时器完全在 C++ 的 libuv 事件循环中运行。

#### 2.2 callSync 机制与诊断工具

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `2e9ccda83` | 2024-03-21 | huangyouhua | 在 `ArkTSTurboModule` 中添加 `callSync()` |
| `008dddb15` | 2024-07-31 | Arkadiusz Kasprzyk | callSync 超过 2ms 时输出 WARNING 日志 |
| `13620ab68` | 2024-09-19 | quxin024 | 为 callSync/runSyncTask 添加 TraceSection |

**架构解读**：`callSync` 是 C++ 侧调用 ArkTS TurboModule 的同步阻塞方式。它通过 `runSyncTask` 在目标线程上执行 ArkTS 方法，调用线程阻塞等待结果返回。

`ArkTSTurboModule.cpp:63-98` 中的实现包含计时和警告逻辑：

```cpp
auto start = std::chrono::high_resolution_clock::now();
m_ctx.taskExecutor->runSyncTask(
    m_ctx.turboModuleThread, [ctx = m_ctx, &methodName, &args, &result]() {
      ArkJS arkJS(ctx.env);
      auto napiArgs = arkJS.convertIntermediaryValuesToNapiValues(std::move(args));
      auto napiResult = napiTurboModuleObject.call(methodName, napiArgs);
      result = arkJS.getDynamic(napiResult);
    });
auto stop = std::chrono::high_resolution_clock::now();
auto duration = std::chrono::duration_cast<std::chrono::milliseconds>(stop - start);
if (duration.count() > 2) {
  LOG(WARNING) << "ArkTSTurboModule::callSync: execution time — "
               << duration.count() << " ms (" + this->name_ + "::" + methodName + ")";
}
```

这个计时器既是诊断工具，也是后续"识别哪些 TurboModule 不应跨线程运行"的发现手段——正是它暴露了 `ImageLoader::getPrefetchResult` 的 3.678ms 耗时问题。

### Phase 2 — TurboModule 线程归属优化（2024-07 ~ 2024-08）

#### 2.3 WorkerTurboModule → AnyThreadTurboModule 架构变迁

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `c79d966c1` | 2024-07-09 | Arkadiusz Kasprzyk | 引入 WorkerTurboModule |
| `ece93ea5d` | 2024-07-26 | Arkadiusz Kasprzyk | AppearanceTM 移至 Worker 线程 |
| `b6c5819a2` | 2024-08-01 | Arkadiusz Kasprzyk | Networking/WebSocket/Blob/FileReader 移至 Worker |
| `3c60951de` | 2024-08-01 | Arkadiusz Kasprzyk | SafeArea/StatusBar 移至 Worker |

**架构解读**：这一阶段将不需要 UI 线程 API 的 TurboModule 移到 Worker 线程，以减轻 UI 线程负载。然而实践发现，**频繁与 UI 线程交互的 TurboModule 放在 Worker 上反而有害**。

`TurboModule.ts` 最终演化为三层分类体系：

```typescript
// UITurboModule — 只能运行在 UI 线程
export class UITurboModule {
  public constructor(protected ctx: UITurboModuleContext) {}
}

// WorkerTurboModule — 已废弃 (latestRNOHVersion: 0.72.38)
// @deprecated Use AnyThreadTurboModule
export class WorkerTurboModule { ... }

// AnyThreadTurboModule — 可在 UI 或 Worker 线程运行
export class AnyThreadTurboModule {
  public constructor(protected ctx: AnyThreadTurboModuleContext) {}
}
```

这个分类的关键洞察：**TurboModule 的线程归属不应由"是否耗时"决定，而应由"与谁通信最频繁"决定**。如果一个 TM 的主要调用者运行在 UI 线程，TM 就应该运行在 UI 线程。

### Phase 3 — 短路执行与 Fabric 优化（2024-08 ~ 2024-09）

#### 2.4 延迟任务在当前线程直接创建

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `2a78c8105` | 2024-08-08 | Juliusz Wajgelt | `runDelayedTask` 在当前线程直接创建定时器 |

**改动细节**（`EventLoopTaskRunner.cpp:61-88`）：

```cpp
DelayedTaskId EventLoopTaskRunner::runDelayedTask(...) {
  auto createTimer = [...]() mutable { ... };
  if (isOnCurrentThread()) {
    createTimer();                    // ← 直接创建，不走 uv_async
  } else {
    runAsyncTask(std::move(createTimer));  // ← 跨线程投递
  }
}
```

`m_nextTaskId` 从 `uint64_t` 改为 `std::atomic<DelayedTaskId>`，支持跨线程并发访问。同样 `cancelDelayedTask` 也做了相同的短路优化。

#### 2.5 Mounting 操作在 UI 线程立即执行

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `b7a58c5d4` / `449f20ab7` | 2024-09-19/21 | Juliusz Wajgelt / bulloo | `performOnMainThread` 增加 `isOnTaskThread(MAIN)` 判断 |

**改动细节**（`SchedulerDelegate.h:86-102`）：

```cpp
template <typename Operation>
void performOnMainThread(Operation operation) {
  if (m_taskExecutor->isOnTaskThread(TaskThread::MAIN)) {
    if (auto mountingManager = m_mountingManager.lock()) {
      operation(mountingManager);     // ← 直接执行
    }
    return;
  }
  m_taskExecutor->runTask(TaskThread::MAIN,
      [weakMountingManager = m_mountingManager, operation = std::move(operation)] {
        if (auto mountingManager = weakMountingManager.lock()) {
          operation(mountingManager); // ← 异步执行
        }
      });
}
```

**关键背景**：Reanimated 在 UI 线程触发 Fabric 更新，导致 `SchedulerDelegate` 回调在 MAIN 线程被调用。优化前这些回调仍会通过 `runTask` 投递回 MAIN 线程，形成一次无意义的"自我投递"。

#### 2.6 Fabric Transaction 拉取移到 MAIN 线程

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `4e363d5db` | 2024-09-30 | Juliusz Wajgelt | `pullTransaction` 从 JS 线程移至 MAIN 线程 |

**改动细节**：旧流程中 `pullTransaction` 在 JS 线程发起，三个回调（willMount / doMount / didMount）分别通过 `performOnMainThread` 投递到 MAIN 线程，transaction 对象需要多次拷贝。

新流程将整个 `pullTransaction` 移到 MAIN 线程执行，三个回调直接在 MAIN 线程上调用，加上 `performOnMainThread` 的短路判断，**整个 mounting 管线在 MAIN 线程上零跨线程通信完成**。

### Phase 4 — ImageLoader 迁移与 BACKGROUND 废弃（2024-10）

#### 2.7 ImageLoaderTurboModule 从 Worker 迁移到 UI 线程

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `d12f7605a` / `9c6d7f207` | 2024-10-10 | Juliusz Wajgelt | 删除 `ImageLoaderWorkerTurboModule.ts`，改用 `UITurboModule` |

**性能数据**（来自 commit message）：

| 场景 | getPrefetchResult 耗时 |
|------|------------------------|
| Worker 线程（跨线程 callSync） | 3.678 ms |
| MAIN 线程（同线程直接调用） | 0.401 ms |
| **性能提升倍数** | **~9x** |

**改动**：
- 删除 `ImageLoaderWorkerTurboModule.ts`（88 行）
- `ImageLoaderTurboModule` 从 `WorkerTurboModule` 改为 `UITurboModule`
- 新增 `getPrefetchResult` 方法直接返回内存缓存结果
- `ImageSourceResolver.h:264-284` 中的 `callSync("getPrefetchResult", ...)` 不再跨线程

**深层原因**：`ImageComponentInstance`（在 MAIN 线程）需要与 `ImageLoaderTurboModule` 通信获取图片预取状态。在 Worker 线程时需要 `runSyncTask` 跨线程阻塞调用，在 MAIN 线程时 `callSync` 的 `runSyncTask` 短路判断使其直接执行。

#### 2.8 BACKGROUND 线程废弃与移除

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `3aeb14a57` | 2024-05-28 | Arkadiusz Kasprzyk | 标记 `enableBackgroundExecutor` 为 @deprecated |
| `1dc0a8efa` | 2024-10-28 | Arkadiusz Kasprzyk | 完全移除 `enableBackgroundExecutor` 功能 |

**废弃原因**（`RNInstance.ts:499-503`）：

```typescript
/**
 * UNSTABLE: Enables an additional BACKGROUND thread for layout calculations.
 * 增加死锁和崩溃风险，通常因 "Assertion failed: parser was already assigned" 错误崩溃。
 * @deprecated: Do not enable this option. (latestRNOHVersion: 0.72.27)
 */
enableBackgroundExecutor?: boolean;
```

`TaskExecutor.h:22-26` 中的 `BACKGROUND` 枚举标记为 `@deprecated`，`TaskExecutor.cpp:27` 中 backgroundTaskRunner 为 `nullptr`。

**架构解读**：BACKGROUND 线程试图将 ShadowTree 布局计算移出 JS 线程，但 ShadowNode 的状态在多线程间需要大量锁保护和断言检查。废弃 BACKGROUND 线程本身就是"减少线程间通信"——通过消除一个线程来消除它带来的所有通信问题。

### Phase 5 — 缓存跨线程调用结果（2026-01）

#### 2.9 SafeAreaView 缓存 insets 减少 TurboModule sync 调用

| 提交 | 日期 | 作者 | 说明 |
|------|------|------|------|
| `c9c7351b7` / `57335d2a5` / `9e2ef73c1` | 2026-01-13 | luqi0119 | 缓存 `SafeAreaTurboModule.getInitialInsets()` 结果 |

**改动细节**（`SafeAreaView.harmony.tsx:35-41`）：

```typescript
let cachedInitialInsets: SafeAreaInsets | null = null;
const getCachedInitialInsets = (): SafeAreaInsets => {
  if (!cachedInitialInsets) {
    cachedInitialInsets = safeAreaTurboModule.getInitialInsets();
  }
  return cachedInitialInsets;
};
```

**意义**：每个 SafeAreaView 实例在 mount 时调用 `getInitialInsets()` 4 次（top/left/right/bottom），每次都是 `callSync` 跨线程调用。缓存后只有第一次跨线程，后续实例直接使用缓存值。当 `SAFE_AREA_INSETS_CHANGE` 事件到来时更新缓存。

## 3. 优化策略总结

| 策略 | 模式 | 典型实现 | 覆盖改动 |
|------|------|----------|----------|
| **短路执行** | 已在目标线程 → 直接调用，跳过 task 投递 | `isOnCurrentThread()` / `isOnTaskThread()` 检查 | Phase 1 (EventLoopTaskRunner)、Phase 3 (performOnMainThread、runDelayedTask) |
| **线程归属优化** | 将 TurboModule 放在与主要调用者同一线程 | ImageLoader → UI; Networking → Worker | Phase 2 (AnyThreadTurboModule)、Phase 4 (ImageLoader) |
| **C++ 替代 ArkTS** | 将原本需要 JS↔ArkTS 跨线程的操作迁至 C++ 单线程 | TimingTurboModule → HarmonyTimerRegistry | Phase 1 |
| **消除线程** | 废弃因线程间通信导致稳定性问题的线程 | BACKGROUND 线程废弃与移除 | Phase 4 |
| **管线合并** | 将多步跨线程投递合并为单步在目标线程上执行 | pullTransaction 整体移至 MAIN 线程 | Phase 3 |
| **结果缓存** | 缓存跨线程调用结果，避免重复 callSync | SafeAreaView insets 缓存 | Phase 5 |

## 4. 稳定性副作用

性能优化往往引入新的稳定性风险，以下是与"减少线程间通信"直接相关的已知问题：

| 问题 | 模块 | 修复 | 根因 |
|------|------|------|------|
| EventBeat::induce 悬空引用 | EventBeat | `b67afac36` (2025-09) / `2df264852` (2025-09) | 析构与 request() 对 `m_unsubscribeUITickerListener` 的跨线程竞争 |
| onAnimationStarted UAF | RNInstanceInternal | `fe72ed2ed` (2025-04) | VSync 回调捕获 `this` 裸指针，实例释放后回调仍执行 |
| onUITick 崩溃 | RNInstanceCAPI | `0b9cc7c19` (2024-08) | VSync 回调中读取已释放实例状态 |
| BACKGROUND 线程死锁/崩溃 | TaskExecutor | `1dc0a8efa` (2024-10) | ShadowNode 多线程状态竞争 |

这些问题的共同特征：**按需订阅/取消 VSync 或 UITick 监听器时，回调可能在线程间竞争到达**。短路执行优化虽然在正常路径上减少了线程间通信，但在边缘场景（对象析构、线程竞争）中引入了新的时序敏感问题。

## 5. 与 React Native 上游的对齐

| RNOH 改动 | 上游对应 | 对齐情况 |
|-----------|----------|----------|
| `EventLoopTaskRunner` (libuv) | iOS 的 `RCTMessageThread` / Android 的 `MessageQueueThread` | RNOH 使用 libuv 事件循环模型，与 iOS 更接近 |
| `BACKGROUND` 线程废弃 | 上游 RFC #744 "Well-defined Event Loop" | 完全对齐，上游也废弃了 BACKGROUND 线程概念 |
| `pullTransaction` 在 MAIN 线程 | iOS 的 `RCTMountingManager.mm:264-284` | 完全对齐，commit message 明确引用了 iOS 实现 |
| `SynchronousEventBeat` / `AsynchronousEventBeat` | Fabric 的 `EventBeat` 体系 | 对齐，但 RNOH 增加了 VSync 对齐层 |
| `AnyThreadTurboModule` | 上游无对应概念 | RNOH 独有，因鸿蒙 Worker 线程模型不同于 iOS/Android |
| `SafeAreaView insets 缓存` | Android 的 `SafeAreaContextCache` | 思路类似 |
